# ADR-0024: 厳格な決定論とクラッシュ駆動開発の強制

## ステータス
提案中 (Accepted)

## コンテキスト
ASMR/ボイスコンテンツ制作において、大気の整合性（Atmospheric Integrity）は極めて繊細なバランスの上に成り立っている。
従来のエンジニアリングにおける「防御的プログラミング（Defensive Programming）」や「暗黙的なフォールバック（Implicit Fallbacks）」は、開発の利便性を高める一方で、品質の劣化（ASR認識率の低下、感情表現の不一致、不自然な間隔など）を隠蔽してしまう副作用があった。

## 決定
以下の原則をエンジン全体に強制する。

1.  **NO FALLBACKS**: 
    - ビジネスロジック内の `try-catch` を禁止する。
    - パース失敗、型不一致、バリデーション失敗時は、デフォルト値を返さず、即座に明示的なエラーをスローしてプロセスを停止（クラッシュ）させる。
2.  **NO HARDCODE**:
    - 実行時の定数（チャンク長、シード、ポーズ時間、リトライ回数等）をソースコード内に固定しない。
    - すべての設定を `0000_config.json` 等の外部入力から取得し、欠落している場合は実行を拒否する。
3.  **DETERMINISTIC FAILURE**:
    - 失敗を「不具合」ではなく「設計へのフィードバック」として扱う。
    - バリデーション（ASR QA等）に失敗した場合、閾値を下げて継続するのではなく、失敗した事実を露呈させ、アイデンティティ設定やスクリプトの修正を促す。
4.  **MECHANICAL COHERENCE & ZERO-TRUST**:
    - すべての実装は `CoherenceAuditor` による機械的な一貫性検証（CoDD）を通過しなければならない。
    - 実行成果物は改ざん不能なハッシュ、マシン生成ログ、および実行トレースを含む `Zero-Trust Contract` によって証拠（Evidence）に基づき検証されなければならない。
    - 設計（ADR）と実装の乖離（Drift）は許容されず、検知された場合は即座に修正を要求する。

## 理由
- **品質の透明性**: ASMRにおける「ささやき」の強度が強すぎる、あるいは滑舌が悪すぎるなどの問題は、ASR認識率に直結する。フォールバックを排除することで、これらの品質低下をリアルタイムで検知できる。
- **アイデンティティの独立性**: 各ボイスアイデンティティ（例：カフカ、メイド）は、それぞれ異なる間（Pause）や共鳴（Resonance）の特性を持つ。ハードコーディングを排除することで、アイデンティティごとの動的なチューニングを保証する。
- **計算リソースの最適化**: 10分以上の長尺音声を生成した後に品質不足が判明するのは非効率である。途中のステップで厳格にクラッシュさせることで、無駄な生成プロセスを最小化する。

## コンシクエンス
- **設定の厳格化**: アセットごとの設定ファイル（`0000_config.json`）の定義が必須となり、設定漏れが許容されなくなる。
- **デバッグの容易性**: 不具合が隠蔽されないため、根本原因（ROOT CAUSE）の特定が容易になる。
- **出力品質の保証**: 完走したプロセスは、すべての品質ゲートを「本質的に」通過したことが保証される。
