# Gemini 設定

最小限の変更で目標を達成してください。不要な変更はバグの原因となります。
コードにコメントは不要です。コード自体で意図を伝えてください。
常に簡潔で明快な日本語を使用してください。かわいい日本語は禁止です。

# KJ哲学：エンジニアリング原則

このプロジェクトでは、コンテキストを「管理された依存関係」として扱います。
プロフェッショナルで決定論的なエンジンとして振る舞ってください。

- **ゼロ・ファット（Zero-Fat）**: `SKILL.md` の指示を厳守してください。
- **ワークフローの隔離**: 調査、設計、計画を分離し、一歩ずつ進めてください。
- **物理的制約**: 4096トークンの文脈窓を意識し、不要な情報は排除してください。

# 安全装置の排除 (No Safety Nets)

防御的なコードは書かないでください。`try-catch` は原則禁止です。
バグは隠さず、クラッシュさせることで原因を特定してください。

- 最小限の実装を行い、即座に実行し、必要であれば破壊してください。
- クラッシュを設計へのフィードバックとして活用してください。
- 根本原因（ROOT CAUSE）を特定し、抜本的に修正してください。

# プロジェクト構造 (Project Structure)

ディレクトリの整合性を維持してください。

- ADR は `docs/adr/README.md` をインデックスとして管理してください。
- `Taskfile.yml` を唯一の CLI エントリポイントとしてください。
- **TS/Bun**: 必ず `bun` を使用してください。
- **Python**: 必ず `uv run` を使用してください。
- `src/domain/*`: ビジネスロジックを集約してください。
- `src/io/*`: 入出力処理を分離してください。
- 設定は `config/default.yaml` に集約してください。

# コード品質 (Code Quality)

- コミット前にリンターと型チェック（`tsc --noEmit` 等）を実行してください。
- データ構造のバリデーションには Zod または Pydantic を使用してください。

# フロントエンド (Frontend)

- シンプルな HTML を優先し、フレームワークの使用は最小限に留めてください。
- `task dev` を使用して開発してください。
